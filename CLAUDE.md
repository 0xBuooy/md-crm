# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

The `md-crm` skill: a personal CRM implemented as a set of agent instructions that drive an LLM to compile freeform interaction notes into an interlinked markdown wiki. There is no runtime code beyond test harness — the "product" is the prompt in `SKILL.md` / `AGENTS.md` / `OPENCODE.md` plus the seed templates under `crm/`.

## Critical: three instruction files are kept byte-identical

`SKILL.md` (Claude Code / Hermes / OpenClaw), `AGENTS.md` (Codex), and `OPENCODE.md` (OpenCode / Pi) must stay in sync — they are the same skill contract shipped under the filename each agent looks for. When editing the skill, update all three together. Verify with:

```bash
diff -q SKILL.md AGENTS.md && diff -q SKILL.md OPENCODE.md
```

## Commands

```bash
# Run both e2e suites on the host (requires hermes + openclaw CLIs + provider API key)
make test

# Same, but in Docker (only requires Docker + ANTHROPIC_API_KEY)
make docker-build       # first time
make test-docker

# One suite at a time
make test-hermes
make test-openclaw
make test-docker-hermes
make test-docker-openclaw

# Filter to a subset of transcripts by filename substring
E2E_FILTER=09-company ./tests/e2e/hermes/run.sh
E2E_FILTER=lint       make test-openclaw

# Verbose step-by-step output
./tests/e2e/hermes/run.sh -v

# Override the agent invocation if your build uses different flags
HERMES_AGENT_CMD="hermes ask --skill md-crm" ./tests/e2e/hermes/run.sh
OPENCLAW_AGENT_CMD="openclaw exec --skill md-crm" ./tests/e2e/openclaw/run.sh

# Disposable sandbox for manual testing (installs into .sandbox/md-crm-e2e/)
make sandbox                 # both tools
make sandbox-hermes          # hermes only
make sandbox-openclaw        # openclaw only
source .sandbox/md-crm-e2e/hermes/env.sh
make sandbox-clean
```

Test dependency: `python3` with `pyyaml` (`pip install pyyaml`).

## Architecture

### Skill contract (the product)

Three operations, exhaustively defined in `SKILL.md`: **ingest**, **query**, **lint**. Query is read-only; lint is read-only; only ingest writes. The skill must never modify files outside `${wiki_path}`, except writing raw notes into a `raw_sources` directory under `${raw_path}`.

### Path resolution

Every operation resolves `wiki_path` and `raw_path` with this precedence (enforced by test `25-config-precedence`):

1. Hermes context — `~/.hermes/config.yaml` → `skills.config.{wiki_path,raw_path}`
2. OpenClaw config — `~/.openclaw/openclaw.json` → `skills.entries.md-crm.config.{wiki_path,raw_path}`
3. `./crm/_config.md` frontmatter — escape hatch for agents without native config (Claude Code, Codex)
4. Defaults — `wiki_path = ./crm`, `raw_path = .`

The repo's own `crm/` directory doubles as both the seed template (copied into a fresh wiki) and the fallback wiki when running Claude Code locally.

### Wiki layout

```
${wiki_path}/
  _config.md        # decay thresholds, raw_sources, stale_thread_days
  index.md          # regenerated by ingest/lint
  people/           # one .md per person, schema in schema/person.md
  companies/        # one .md per company, schema in schema/company.md
  interactions/log.md   # reverse-chronological global log
  schema/{person,company}.md
```

Principle: the wiki is *compiled* knowledge. Queries hit compiled pages, not raw notes. Contradictions are flagged with `> [!warning]` callouts, never silently overwritten.

### E2E test harness

`tests/e2e/` drives real agents through YAML transcripts and asserts on filesystem state, not agent prose (prose is non-deterministic across model revisions).

- `tests/e2e/fixtures/*.yaml` — one transcript per scenario (25 scenarios, numbered 01–25 by concern)
- `tests/e2e/lib/run_transcript.py` — loops over fixtures, wipes wiki + raw between each
- `tests/e2e/lib/validate.py` — YAML-frontmatter / body / glob matcher
- `tests/e2e/{hermes,openclaw}/run.sh` — provisions an isolated `HERMES_HOME` / OpenClaw config pointing at a temp `wiki_path` + `raw_path`, installs the skill from the checkout, invokes the runner

Assertion philosophy: **structural keys exact, prose fuzzy.** Slugs, frontmatter field names, and wiki-links are checked literally; role/company strings match case-insensitive substring so "head of partnerships" ≡ "Head of Partnerships" doesn't flake. Supported matchers: `equals`, `contains`, `regex`, `any_of` — see `tests/e2e/README.md` for the full YAML schema.

### Writing a new e2e scenario

Add one YAML to `tests/e2e/fixtures/`. Keep each file focused on a single behavior — that's why there are 25 small files instead of a few omnibus ones. Fixture schema and matcher reference live in `tests/e2e/README.md`.
